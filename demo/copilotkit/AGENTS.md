# AGENTS.md — demo/copilotkit/

## Overview

**PlatformOps** — a Next.js 16 internal developer portal where the primary interface is conversation. Demonstrates all three rendering families (MCP Apps, A2UI, AG-UI streaming) flowing through one CopilotKit chat against one MCP server.

The longer README in this directory explains *what* the IDP does. This file is the operational guide for changing it.

## Setup

```bash
cp .env.example .env.local   # then fill in ANTHROPIC_API_KEY
pnpm install
pnpm dev                     # http://localhost:3002
```

Start `../` (the `demo/` MCP server) **first** — without it the catalog and MCP App UI are missing.

## Commands

| Task | Command | Notes |
|------|---------|-------|
| Install | `pnpm install` | pnpm, not npm — `pnpm-lock.yaml` is authoritative |
| Dev | `pnpm dev` | Next.js dev server on port **3002** |
| Build | `pnpm build` | production build |
| Start | `pnpm start` | run production build on port 3002 |
| Typecheck | `pnpm typecheck` (alias `pnpm lint`) | `tsc --noEmit` — there is no eslint here |

`pnpm-lock.yaml` pins `rxjs@7.8.1` via overrides because CopilotKit and AG-UI disagree. Don't remove the override.

## Environment

| Var | Required | Default | Notes |
|-----|----------|---------|-------|
| `ANTHROPIC_API_KEY` | yes | — | Sonnet 4.5+ access needed |
| `MCP_SERVER_URL` | no | `http://localhost:3001/mcp` | points at the `/demo` MCP server |

`.env.local` is gitignored and **must never be committed**. Only `.env.example` lives in git.

The `/demo` server (this repo's `demo/`) must be running before `pnpm dev` will be useful — the catalog data and MCP App UI come from there.

## Code style / Conventions

- TypeScript strict, React 19, Next.js 16 App Router, ESM.
- No ESLint config — `tsc --noEmit` (aliased as both `pnpm lint` and `pnpm typecheck`) is the only static check.
- CSS via Tailwind v4 (`@tailwindcss/postcss`). Global styles in `app/globals.css`.
- State via **zustand** (`components/store.ts`) — no Redux, no Context for shared state.
- Use the **v2** CopilotKit API surface (`@copilotkit/react-core/v2`). v1 imports are legacy.

## The three rendering families (how this app wires them)

| Family | Where it's registered | What it renders |
|--------|----------------------|-----------------|
| **MCP Apps** | `MCPAppsMiddleware` in `route.ts` + `mcpAppsCanvasRenderer` in `App.tsx` | Full HTML resources from the `/demo` MCP server, embedded as iframes in the canvas |
| **A2UI** | `A2UIMiddleware({ injectA2UITool: true })` in `route.ts` + `a2uiCanvasRenderer` + `platformA2UICatalog` | Declarative JSON UI generated by the model |
| **AG-UI streaming** | `useFrontendTool(...)` in `DeployStatus`, `StreamActionTool`, `ChartTool`, `DataTools` | Live in-canvas tool results, streamed phase-by-phase |

If you add a new capability, decide **first** which family it belongs to:
- Need a durable, designed page? → **MCP App** in `/demo/server.ts` + the catalog
- Need a generated work surface (per-resource board, comparison view)? → **A2UI** with an entry in `platformA2UICatalog.tsx`
- Need a live operational action with progress? → **AG-UI streaming** via `useFrontendTool` in a new component under `components/`

## Workflow

1. Start `/demo` first (`cd ../.. && cd demo && npm run start`), then `pnpm dev` here.
2. Edit, save — Next dev rebuilds. Hot reload works for components; route handlers in `app/api/` require a hard refresh.
3. `pnpm typecheck` before claiming done — there is no other static check.
4. **Open the running app at `http://localhost:3002` and exercise the workflow you changed.** Type-check passing does not prove the chat → middleware → canvas path works.

## Boundaries

### Always Do
- Use CopilotKit v2 APIs (`@copilotkit/react-core/v2`). The legacy v1 API surfaces are still exported but the talk shows v2.
- When adding a `useFrontendTool`, register a matching activity renderer in `<CopilotKitProvider renderActivityMessages={[...]}>` so the tool result reaches the canvas.
- Keep `route.ts` middleware order stable: `MCPAppsMiddleware` then `A2UIMiddleware`. Reordering changes which tool wins for ambiguous renders.

### Ask First
- Bumping `@copilotkit/*`, `@ag-ui/*`, `@a2ui/web_core`, `next`, `react`, or `@modelcontextprotocol/ext-apps`. All of these are pre-1.0 or just-1.x and break across minor versions.
- Adding a server-side persistence layer — the demo is intentionally stateless beyond the zustand store.
- Changing port `3002` — referenced in the talk and in `.env.example` comments.

### Never Do
- Commit `.env.local` (or any file containing an Anthropic key).
- Commit `.next/`, `out/`, `*.tsbuildinfo`, or `next-env.d.ts` (all gitignored).
- Remove the `outputFileTracingRoot` pin in `next.config.ts` — without it Next traces into `../package-lock.json` and tries to bundle the wrong deps.
- Replace pnpm with npm/yarn here.

## Security

- `ANTHROPIC_API_KEY` lives in `.env.local` (gitignored). It is read server-side in `app/api/copilotkit/[[...slug]]/route.ts` and never exposed to the browser.
- `app/api/mcp-resource/route.ts` proxies arbitrary `ui://` URIs from the configured MCP server. Keep `MCP_SERVER_URL` pointed at trusted hosts — never accept it from the client.
- Treat the Anthropic key as production-grade: any leak invalidates it. Do not log request bodies or model output to disk.

## Examples / Patterns to Follow

Adding a streaming frontend tool — register the hook in a component and the renderer in `App.tsx`:

```tsx
// components/MyAction.tsx
useFrontendTool({
  name: "my-action",
  description: "Run my-action against a resource",
  parameters: { /* zod schema */ },
  render: ({ status, args }) => <MyActionCard status={status} {...args} />,
});

// components/App.tsx
<CopilotKitProvider
  runtimeUrl="/api/copilotkit"
  renderActivityMessages={[mcpAppsCanvasRenderer, a2uiCanvasRenderer /*, myActionRenderer */]}
>
```

Existing `DeployStatus.tsx`, `StreamActionTool.tsx`, and `ChartTool.tsx` are golden samples — copy their shape rather than improvising.

## When stuck

- If a tool fires but nothing appears in the canvas, check that an activity renderer for that tool name is registered in `App.tsx`.
- If the canvas shows an MCP App iframe error, the `/demo` server probably isn't running, or `MCP_SERVER_URL` is misconfigured.
- If A2UI renders blank, check that the component the model emitted exists in `platformA2UICatalog.tsx`.
